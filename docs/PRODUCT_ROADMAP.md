# WB analyst: план развития продукта

Документ фиксирует план развития после релиза `v1.0`. Цель - двигаться не хаотично, а выпускать по одной полноценной функции: спроектировали, реализовали, проверили, довели до стабильности, выпустили в `main`, после финальной проверки продублировали в `prod`.

## 1. Текущее состояние v1.0

Уже есть сильная база:

- загрузка финансовых отчетов WB через Finance API;
- поддержка WB-периодов и ручного диапазона;
- загрузка рекламы через Promotion API;
- rate limiter и повторы при лимитах Finance API;
- локальный кэш по периоду, типу отчета и отпечатку API-токена;
- ручная активация;
- себестоимость товаров;
- расходы периода;
- налоговые настройки;
- пересчет уже загруженного отчета без повторной загрузки из WB;
- Excel-экспорт;
- разделы `Автоотчет`, `Себестоимость товаров`, `Инструкция`, `Логи`;
- GitHub-репозиторий с ветками `main` и `prod`;
- landing-сайт в папке `landing`;
- релизный ZIP в `releases/WB-analyst-v1.0.zip`.

Главная ценность v1.0: программа отвечает не только на вопрос "сколько продаж", а на вопрос "сколько денег реально осталось после WB, рекламы, налогов, себестоимости и расходов".

## 2. Позиционирование продукта

WB analyst должен развиваться не как "генератор Excel", а как кабинет управленческой аналитики для малого и среднего WB-селлера.

Ключевые вопросы, на которые продукт должен отвечать:

- сколько я реально заработал;
- какие товары приносят прибыль;
- какие товары съедают прибыль;
- где WB удержал деньги и за что;
- какая реклама работает в плюс, а какая в минус;
- какие товары стоит закупать дальше;
- что изменится, если поменять цену, себестоимость, рекламу или налог;
- где у бизнеса риски: возвраты, логистика, хранение, реклама, кассовый разрыв.

## 3. Правило разработки новых версий

Каждая новая версия должна проходить один и тот же цикл:

1. Описание функции: что добавляем и зачем селлеру.
2. Источники данных: какие таблицы, API, кэш, настройки нужны.
3. Расчетная логика: формулы, поля, ограничения, спорные случаи.
4. UI-макет: где функция появляется, какие состояния есть.
5. Реализация.
6. Автотесты на расчетную логику.
7. Ручные тесты на реальных отчетах.
8. Проверка Excel-экспорта, если функция попадает в Excel.
9. Проверка кэша, пересчета и повторной загрузки.
10. Сборка exe.
11. Обновление landing/README/инструкции.
12. Push в `main`.
13. Финальная ручная проверка.
14. Push/merge в `prod` и тег версии.

Минимальная проверка перед каждым релизом:

```bash
cd wb_finance_analyst
python -m compileall .
python -m pytest
python main.py
```

Для landing:

```bash
cd landing
npm start
```

### v1.2 - Дашборд владельца

Цель: первый экран после загрузки отчета должен сразу объяснять состояние бизнеса.

Что добавить:

- блок "Что хорошо";
- блок "Что плохо";
- блок "Где потеряли деньги";
- топ прибыльных товаров;
- топ убыточных товаров;
- топ удержаний WB;
- предупреждения по качеству данных:
  - нет себестоимости;
  - реклама не загружена;
  - есть большой возврат;
  - много логистики;
  - хранение выросло;
  - отчет загружен не за полный WB-период.

UI:

- сверху компактные фильтры;
- ниже карточки метрик;
- затем блок выводов;
- затем вкладки с таблицами.

Критерий готовности:

- селлер после загрузки отчета за 10-20 секунд понимает, прибыльный период или нет, и почему.

### v1.3 - Прибыльность товаров как главный продуктовый отчет

Цель: сделать вкладку/отчет "Прибыль по товарам" центральной ценностью продукта.

Поля:

- артикул поставщика;
- название товара;
- продано;
- возвраты;
- выкуп, %;
- к перечислению за товар;
- логистика;
- хранение;
- реклама;
- себестоимость;
- упаковка;
- налог;
- чистая прибыль;
- маржинальность;
- статус: прибыльный / риск / убыточный.

Важные правила:

- основной ключ товара - артикул поставщика;
- название товара хранить отдельно как справочное поле;
- продажи на вкладке `Продажи` можно дополнительно группировать по цене WB, чтобы видеть разные цены одного товара;
- в товарной прибыли считать итог по товару целиком, а не дробить по цене, если это не нужно для решения.

Критерий готовности:

- можно открыть один отчет и понять, какие товары закупать, какие пересмотреть, какие убрать.

### v1.4 - Реклама WB и ДРР

Цель: отделить "просто расход на рекламу" от управляемой рекламной аналитики.

Что добавить:

- отчет по рекламным кампаниям;
- расход по кампаниям;
- связь кампаний с товарами, если API позволяет получить артикулы;
- ДРР;
- прибыль после рекламы;
- кампании в минус;
- кампании без продаж;
- понятная пометка, если реклама не может быть точно привязана к товару.

Правило для v1.x:

- если выбран тип отчета `Оба`, рекламу вычитать целиком один раз;
- если выбран один тип отчета, распределять рекламу пропорционально `Итого к оплате WB` и указывать это в обозначениях.

Критерий готовности:

- пользователь видит не только сумму рекламы, но и понимает, где реклама работает плохо.

### v1.5 - История отчетов и сравнение периодов

Цель: превратить кэш в понятную историю, а отчеты - в динамику.

Что добавить:

- экран `История отчетов`;

- дата загрузки;
- тип отчета;
- кабинет/отпечаток токена;

Показатели сравнения:

- себестоимость;

- маржинальность;
- возвраты;

- хранение.

Критерий готовности:

## 6. Технические принципы

### Данные и кэш

- кэш не должен смешивать разные кабинеты;
- ключ кэша должен учитывать источник, период, тип отчета, выбранные WB-периоды и отпечаток токена;
- новые версии кэша должны иметь версию схемы;
- при изменении структуры данных WB нужно сохранять обратную совместимость там, где это возможно.

### API

- все Finance-запросы должны идти через единый клиент;
- не делать лишние проверки токена перед загрузкой;
- уважать лимиты WB API;
- 429 не считать критической ошибкой до исчерпания повторов;
- не логировать API-токены;
- логировать endpoint, status code, попытку, retry seconds.

### Расчеты

- расчетная логика должна быть отдельно от UI;
- на каждую новую формулу добавлять тесты;
- спорные поля WB фиксировать в документации;
- Excel и UI должны брать данные из одного результата расчета.

### UI

- интерфейс должен выглядеть как рабочий кабинет аналитики, а не как генератор таблиц;
- карточки - только для ключевых метрик;
- таблицы должны быть фильтруемыми и удобными для проверки;
- пустые состояния должны объяснять, что делать дальше;
- ошибки должны быть понятными для пользователя, технические детали - в логах.

## 7. Релизная схема

Рабочий поток:

1. Разработка идет в `main`.
2. После реализации функции запускаются тесты.
3. Собирается exe.
4. Обновляется `releases/WB-analyst-vX.Y.zip`.
5. Обновляется landing.
6. Push в `main`.
7. Ручная проверка.
8. После подтверждения - `prod` обновляется до этого же коммита.
9. Создается тег `vX.Y`.

Для v1.0 обе ветки уже указывают на один коммит.

## 8. Рекомендуемый ближайший порядок

Ближайшие 5 шагов:

1. `v1.1` - релизная упаковка: GitHub Releases, версия в приложении, "О программе", проверка чистой Windows.
2. `v1.2` - дашборд владельца и блок выводов.
3. `v1.3` - усиленная прибыльность товаров.
4. `v1.4` - реклама WB и ДРР.
5. `v1.5` - история отчетов и сравнение периодов.

Почему так:

- сначала повышаем доверие и удобство установки;
- затем делаем первый экран продающим;
- потом усиливаем самый ценный отчет - прибыль по товарам;
- после этого добавляем рекламу и сравнение периодов как платные преимущества.
